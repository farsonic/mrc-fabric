# Architecture

## Why a persistent agent, not per-app

The OCP MRC 1.0 spec defines `mrc_ctl.h` as a control-plane between an endpoint
and a fabric controller — not a one-shot library call. The profile is expected
to change over the flow's lifetime (re-steering, mode changes, path denies).
Implementing that in every app would re-do the same three jobs:

  1. Generate traffic (the app's actual purpose)
  2. Speak to the controller (cross-cutting)
  3. Own the dataplane (cross-cutting)

A single host-side agent unbundles those. Apps stay MRC-unaware — they open a
plain socket to a peer tenant address. The agent is the MRC endpoint.

## Why a dummy `mrc0` interface

Linux already has every piece needed (seg6 encap routes on the underlay), so
the agent could just program routes on `eth1` directly. The dummy `mrc0`
interface buys three things:

  - **Stable, named "MRC surface."** The host's IPv6 tenant address lives on
    `mrc0`, not the underlay. The identity is decoupled from the physical NIC.
  - **One place to look.** `ip -6 route show dev mrc0` (or scoped queries
    against the tenant address) shows the current EV state without the noise
    of underlay routes.
  - **Future-proofing.** Promoting `mrc0` to a VRF (or a real tunnel device, or
    a kernel module) keeps the same external surface — apps still send to the
    peer's tenant address.

The agent owns the dummy's lifecycle: create on start, address-replace
idempotently, remove on `--delete-iface-on-exit`.

## SSE, not polling

The controller pushes profile changes to subscribed agents via Server-Sent
Events (`/api/profile/stream`). Each agent holds one long-lived connection.
The controller fans out to all subscribers on any state change. Sub-second
push latency from a GUI toggle is normal.

Why SSE and not WebSockets: SSE is one-way (server → client), text-only,
trivially proxyable, works through any HTTP stack with no extra protocol, and
is exactly what we need — the agent's reverse channel is HTTP POST to
`/api/metrics`.

## Why the switches are static

In the lab, the switches are programmed once from `lab/switch-config/`. The
agent on each host changes which uSID stack is encapped per flow; the
switches just forward whatever they see. This matches a real production model:
the fabric is provisioned by a network team and stays put; the **endpoint**
chooses the path per flow.

The static config is generated by the upstream Apache-2.0
[segment-routing/srv6-mrc-emulator](https://github.com/segment-routing/srv6-mrc-emulator);
`controller/gen_switch_config.py` is a thin wrapper that delegates to it if
present, otherwise points at the configs already committed in this repo.

## Metrics

The agent samples `/sys/class/net/<iface>/statistics/{tx,rx}_bytes` once a
second, converts to bps, and POSTs `{host, eth_tx_bps, eth_rx_bps, routes}` to
the controller. The controller stores the latest per host and overlays it on
the access links in `/api/telemetry`. Spine-link bps remains a modeled overlay
(offered-rate / number-of-active-paths) until switch-counter polling is added —
that's the obvious next step on the controller side.
